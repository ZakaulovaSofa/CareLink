## UC-01: Регистрация пользователя в системе

**US-01:**
Я, как волонтёр / координатор, хочу зарегистрироваться в системе с указанием своей роли, чтобы получить персональный доступ к функциональности и данным, связанным с моей деятельностью

| Поле | Значение |
|---|---|
| ID | UC-01 |
| Название | Регистрация пользователя в системе |
| Акторы | Волонтёр / координатор |
| Предусловия | - У пользователя есть доступ к интернету<br>- Пользователь имеет действительный адрес электронной почты<br>- Сервер ИС работает в штатном режиме<br>- Пользователь не зарегистрирован с этой ролью |
| Постусловия (успех) | - Пользователь зарегистрирован в системе с выбранной ролью<br>- Запись о пользователе добавлена в БД |
| Постусловия (неудача) | - Пользователь не зарегистрирован в системе<br>- В БД не добавлена запись о пользователе |
| Основной поток | 1. Пользователь открывает ИС.<br>2. Пользователь выбирает свою роль (волонтёр / координатор).<br>3. Пользователь вводит свою электронную почту.<br>4. Система отправляет письмо‑подтверждение на почту.<br>5. Пользователь переходит по уникальной ссылке в письме (действие ссылки ограничено по времени).<br>6. Пользователь вводит пароль (минимум 8 символов, включают заглавные, строчные буквы и цифры).<br>7. Пользователь повторяет свой пароль.<br>8. Данные о пользователе сохраняются в БД.<br>9. Пользователь получает доступ к ИС. |
| Альтернативные потоки | - Почта уже используется. Точка входа: после шага 3.<br>&nbsp;&nbsp;3.1 Сообщение "Почта уже зарегистрирована в ИС".<br>&nbsp;&nbsp;3.2 Система предлагает пользователю перейти на страницу авторизации.<br>&nbsp;&nbsp;3.3 Кнопки "Войти" / "Вернуться к регистрации".<br><br>- Ссылка подтверждения не действительна. Точка входа: шаг 5.<br>&nbsp;&nbsp;5.1 Сообщение: "Ссылка подтверждения не действительна".<br>&nbsp;&nbsp;5.2 Система предлагает пользователю повторно отправить письмо со ссылкой на почту.<br>&nbsp;&nbsp;5.3 Кнопки "Отправить письмо повторно" / "Вернуться к регистрации".<br><br>- Пароль некорректный. Точка входа: шаг 6.<br>&nbsp;&nbsp;6.1 Сообщение "Пароль должен содержать минимум 8 символов, которые должны включать цифры, строчные и прописные буквы латинского алфавита".<br>&nbsp;&nbsp;6.2 Возвращение к шагу 6 (ввод пароля).<br><br>- Пароли не совпадают. Точка входа: шаг 7.<br>&nbsp;&nbsp;7.1 Сообщение "Пароли не совпадают".<br>&nbsp;&nbsp;7.2 Поля паролей очищаются.<br>&nbsp;&nbsp;7.3 Возвращение к шагу 6 (ввод пароля). |
| Исключительные потоки | - Проблема с доставкой письма‑подтверждения. Точка входа: шаг 5.<br>&nbsp;&nbsp;5.1 Система отправляет письмо, но пользователь его не получает.<br>&nbsp;&nbsp;5.2 Кнопки: "Отправить письмо повторно" / "Изменить адрес электронной почты". |
| Бизнес‑правила | BR-01<br>

**Критерии приёмки в UC-01:**<br>- Пользователь может выбрать одну из ролей: волонтёр / координатор.<br>- Система проверяет уникальность email для выбранной роли.<br>- Пользователь получает письмо‑подтверждение с уникальной ссылкой, действительной в течение 1 часа.<br>- После перехода по ссылке пользователь должен создать пароль.<br>- В БД создана новая запись пользователя с указанной ролью и статусом "Подтверждён".<br>- Пользователь получает доступ к системе с функциональностью, соответствующей его роли.<br>- При повторной регистрации для другой роли система позволяет создать новую роль для того же email (BR-01).<br>- В случае ошибки подтверждения письма система уведомляет пользователя и позволяет запросить повторную ссылку. |

## UC-02: Авторизация пользователя в системе

**US-02:**
Я, как зарегистрированный пользователь, хочу авторизоваться в системе под своей учётной записью, чтобы получить доступ к системе в соответствии со своей ролью

| Поле | Значение |
| --- | --- |
| ID | UC-02 |
| Название | Авторизация пользователя в системе |
| Акторы | Зарегистрированный пользователь (волонтёр / координатор / модератор / администратор) |
| Предусловия | - Пользователь зарегистрирован в системе с выбранной ролью- У пользователя есть интернет-соединение- Сервер ИС работает в штатном режиме- Пользователь не заблокирован модератором |
| Постусловия (успех) | - Пользователь получает доступ к интерфейсу ИС, соответствующему его роли |
| Постусловия (неудача) | - Пользователь не получает доступ к ИС- Пользователь возвращается к авторизации / регистрации |
| Основной поток | 1\. Пользователь заходит в ИС.2. Пользователь переходит на страницу авторизации.3. Пользователь выбирает роль.4. Пользователь вводит почту и пароль.5. Пользователь получает доступ к системе для своей роли. |
| Альтернативные потоки | - Незарегистрированная почта. Точка входа: шаг 4. 4.1 Сообщение "Данная почта не зарегистрирована в системе". 4.2 Кнопки "Зарегистрироваться" / "Указать другую почту". 4.3 Переход к UC-01 / возврат к шагу 4.- Неверный пароль. Точка входа: шаг 4. 4.1 Сообщение "Неверный пароль". 4.2 Кнопки "Ввести пароль заново" / "Восстановить пароль". 4.2.1 ("Ввести пароль заново") Возврат к шагу 3 (выбор роли). При 5 неверных попытках ввода пароль аккаунт блокируется на 72 часа. 4.2.2 ("Восстановить пароль") Система отправляет письмо с уникальной ссылкой (действительна 1 час). 4.2.3 Пользователь переходит по ссылке. 4.2.4 Пользователь задаёт новый пароль по правилам BR-04. 4.2.5 При корректном пароле возврат к шагу 4, иначе сообщение о требованиях к паролю и повторный ввод. |
| Исключительные потоки | - Проблема с доставкой письма-подтверждения (в альтернативном потоке восстановления пароля). Точка входа: шаг 4.2.2. 4.2.2.1 Письмо отправлено, но не доставлено пользователю. 4.2.2.2 Кнопки: "Отправить письмо повторно" / "Изменить адрес электронной почты". |
| Бизнес-правила | BR-04 (упомянуто в потоке восстановления пароля) |

**Критерии приёмки в UC-02:**

-   Пользователь может выбрать роль на странице авторизации;

-   Система проверяет корректность введённых email и пароля;

-   При 5 неверных попытках ввода пароля аккаунт блокируется на 72 часа;

-   При восстановлении пароля пользователь получает письмо с уникальной ссылкой (действует 1 час);

-   В случае ошибки подтверждения система позволяет запросить повторную ссылку;

-   После успешной авторизации пользователь получает доступ к интерфейсу ИС, соответствующему его роли.

## UC-03: Отображение интерфейса в соответствии с выбранной ролью

**US-03:**
Я, как пользователь системы, хочу видеть только те разделы и функции, которые разрешены моей ролью, чтобы работать в системе безопасно и без доступа к лишним для меня данным

| Поле | Значение |
| --- | --- |
| ID | UC-03 |
| Название | Отображение интерфейса в соответствии с выбранной ролью |
| Акторы | Авторизованный пользователь (волонтёр / координатор / модератор / администратор) |
| Предусловия | - У пользователя есть интернет-соединение- Сервер ИС работает в штатном режиме- Пользователь успешно авторизован в системе под выбранной ролью |
| Постусловия (успех) | - Пользователь видит интерфейс ИС, соответствующий его роли- Пользователь может использовать доступную ему функциональность |
| Постусловия (неудача) | - Пользователь видит некорректный интерфейс (лишние разделы или отсутствие нужных) |
| Основной поток | 1\. Пользователь успешно авторизовался в системе.2. Система формирует интерфейс с доступными разделами и функциями в соответствии с ролью.3. Пользователь видит интерфейс системы, соответствующий его роли: 3.1 Волонтёр: "Личный кабинет" (анкета, данные, опыт, история активности, уровень допуска, заявки), "Обучение" (входной курс и тестирование), "Заявки" (доступные задачи, отклик, коммуникация с координатором; при отсутствии допуска — пусто и кнопка "Пройти обучение"). 3.2 Координатор: "Личный кабинет" (активные заявки, связь с волонтёрами, информация об организациях), "Заявки" (история, создание заявки, отправка на модерацию). 3.3 Модератор: "Личный кабинет", "Отчётность", "Волонтёры", "Координаторы", "Заявки" (фильтрация, модерация). 3.4 Администратор: "Панель управления", "Пользователи и роли", "Заявки", "Модерация и контроль", "Контент", "Отчётность", "Настройки системы". |
| Альтернативные потоки | - Не задана или некорректна роль при формировании интерфейса. Точка входа: шаг 2. 2.1 Система не может применить конфигурацию роли. 2.2 Пользователь перенаправляется на страницу ошибки или получает сообщение с предложением перезагрузить систему / обратиться в поддержку. |
| Исключительные потоки | - Ошибка загрузки конфигурации интерфейса. Точка входа: шаг 2. 2.1 Система не смогла применить настройки роли. 2.2 Пользователь видит сообщение об ошибке и ссылку на поддержку. |
| Бизнес-правила | BR-02 |

**Критерии приёмки в UC-03:**

-   После авторизации пользователь видит только разделы, соответствующие его роли;

-   Пользователь не может получить доступ к запрещённым разделам даже при прямом переходе по ссылке;

-   Интерфейс корректно обновляется при смене роли или уровня допуска;

-   Отображение функциональности строго соответствует описанию ролей (волонтёр / координатор / модератор / администратор).

## UC-04: Отображение личного кабинета волонтёра

**US-04:**
Я, как волонтёр, хочу видеть свою анкету с личными данными и подтверждёнными навыками, чтобы понимать, как система и координаторы оценивают мой опыт и знания

| Поле | Значение |
| --- | --- |
| ID | UC-04 |
| Название | Отображение личного кабинета волонтёра |
| Акторы | Волонтёр |
| Предусловия | - Волонтёр имеет доступ к интернету- Сервер работает в штатном режиме- Волонтёр успешно зарегистрирован и авторизован в системе |
| Постусловия (успех) | - Волонтёр видит свой личный кабинет с анкетой, уровнем допуска, историей участия в заявках и личными данными |
| Постусловия (неудача) | - Волонтёр не видит личный кабинет- Волонтёр не может отследить текущие заявки- Волонтёр не знает уровень допуска- Система отображает сообщение об ошибке- Система предлагает обратиться в техподдержку |
| Основной поток | 1\. Волонтёр авторизовался в системе.2. Волонтёр переходит в раздел «Личный кабинет».3. Система отображает: уровень допуска, анкету (доступна для редактирования), историю работы по заявкам, текущие заявки.4. Волонтёр может редактировать анкету и контактные данные. |
| Альтернативные потоки | - Уровень допуска не определён. Точка входа: шаг 3. 3.1 Волонтёр не прошёл входное тестирование. 3.2 Отображается статус «не определён». 3.3 Показано уведомление с предложением пройти обучение.- Уровень допуска на модерации. Точка входа: шаг 3. 3.1 Тестирование пройдено, но не проверено модератором. 3.2 Отображается статус «на модерации». 3.3 Сообщение о скорой проверке результатов. |
| Исключительные потоки | - Ошибка загрузки интерфейса личного кабинета. Точка входа: шаг 2. 2.1 Сообщение «Личный кабинет недоступен». 2.2 Кнопка «Обратиться в поддержку».- Ошибка БД при изменении анкеты. Точка входа: шаг 4. 4.1 Сообщение об ошибке изменения данных. 4.2 Кнопка «Обратиться в поддержку». |
| Бизнес-правила | BR-03, BR-04 |

**Критерии приёмки в UC-04:**

-   Волонтёр имеет доступ к разделу «Личный кабинет»;

-   Отображаются только персональные данные пользователя;

-   Доступно редактирование анкеты;

-   Отображается актуальный уровень допуска;

-   Отображается история и текущие заявки;

-   Отображаются статусы откликов на заявки.

## UC-05: Отображение уровня допуска волонтёра

Я, как волонтёр, хочу видеть свой текущий уровень допуска, чтобы понимать, к каким заявкам у меня есть доступ на данный момент

| Постусловия (успех) | - Волонтёр видит уровень допуска- Волонтёр понимает правила доступа к заявкам |
| Постусловия (неудача) | - Уровень допуска не отображается- Волонтёр не понимает доступные заявки- Система сообщает о недоступности данных- Предлагается обратиться в поддержку |
| Основной поток | 1\. Волонтёр вошёл в личный кабинет.2. Отображается поле «Уровень допуска».3. Поле заполнено.4. Отображается пояснение уровня допуска и доступных заявок.5. Пользователь может ознакомиться со справкой по уровням допуска. |
| Альтернативные потоки | - Уровень допуска не определён. Точка входа: шаг 3. 3.1 Тестирование не пройдено. 3.2 Статус «не определён». 3.3 Предложение пройти обучение.- Уровень допуска на модерации. Точка входа: шаг 3. 3.1 Тест на проверке. 3.2 Статус «на модерации». 3.3 Сообщение о проверке. |
| Исключительные потоки | - Ошибка загрузки уровня допуска. Точка входа: шаг 3. 3.1 Поле пустое. 3.2 Сообщение об ошибке отображения. 3.3 Кнопка «Обратиться в поддержку». |

**Критерии приёмки в UC-05:**

-   В личном кабинете всегда отображается уровень допуска;

-   Уровень допуска сопровождается пояснением;

-   Данные соответствуют текущему статусу пользователя;

-   Доступна справочная информация по уровням допуска.
